What is OAuth.io?

OAuth.io is an API for 80+ OAuth providers and our mission is to alleviate OAuth fragmentation in a practical way for developers.
Technically, OAuth.io is a hosted version of oauthd, the open source OAuth daemon we started developing. The OAuth.io service runs on Amazon EC2 and provides an easy and simple web interface for developers to configure and consume their various OAuth providers in a unified way.

What is OAuth?

OAuth is an open standard for authorization. OAuth provides a method for clients to access server resources on behalf of a resource owner (such as a different client or an end-user). It also provides a process for end-users to authorize third-party access to their server resources without sharing their credentials (typically, a username and password pair), using user-agent redirections.

Wikipedia, OAuth

In practice, influential web companies could not wait for the IETF to finish the OAuth specification (whichever the version). This resulted in a huge fragmentation not only regarding the official versions of the protocol but also regarding implementations that simply are not compliant with any of the specifications. Hence developers pain.

What is the OAuth.io flow?

Your application calls OAuth.popup or OAuth.redirect in javascript. It does a request to oauth.io with the provider to connect.
Then OAuth.io sends the infos to popup/redirect to the provider's authorization form.
Once authorized, the provider redirects to oauth.io which will request the token and send back to the js callback in OAuth.popup / OAuth.callback
In a server-side flow, your application just receive a code that he send (in an ajax request) to his server.
Then the server requests oauth.io with his secret key to exchange the code for the token.

Why do you store my private keys?

OAuth.io is an OAuth backend as a service for your applications so that’s why we ask APIkeys of applications, like other Backend-as-a-Service (BaaS).

How secure is it?

Security is our highest priority in our mission to solve OAuth integration for developers. Security with performance and developer friendly experience.

OAuth.io is an OAuth backend-as-a-service for your applications so that’s why we ask APIkeys of applications, like other Backend-as-a-Service (BaaS).

What we can tell you about how we manage security on OAuth.io:

  • Everything is SSL EV encrypted, to secure the communitcation between oauth.io and the user.
  • - The code is single use and with a short expiration time, so no third person will be able to re-use it
  • - The valid domains/url depend on the detail level: the user can restrict to a single scheme/domain/port/path, so that the redirection cannot be cheated to leak an access token.
  • - The "state" parameter is now mandatory when doing server-side auth, to avoid CSRF exploit.
  • - The client-side/server-side (token/code) is decided in the app configuration, and not in the request, so that getting a token is not as easy as changing a request parameter.
  • - We do not store any access_token to assure privacy in worst attack cases.
  • - We have avoided url fragment leak (to get an access_token from a referer), a reason from recent facebook hacks.
  • - We checked lots of CSRF/XSS potential security exploits and what we have done with Angular.js passed with success all tests
  • - and globally, the website is API based/driven, so its security is easier to manage.

    • To be completely transparent for users which want to host their APIkeys, the code is open source and they can install the open source version of the daemon, called oauthd and accessible on Github https://github.com/oauth-io/oauthd

    What is AGPL and what we can do with this open source licence?

    oauthd is open source under Affero GPL license, the “GPL license for servers”, like MongoDB for example.

    Because oauthd is a stand alone running application as a daemon, so by installing oauthd, users have just to redistribute modifications they have made of oauthd under the same open source licence, and not all the code of the project as other classical GPL license.

    That means:

    • - all open source projects can use oauthd for their apps and projects.
    • - closed source projects using oauthd as a stand-alone server needs to redistribute in open source only modifications of oauthd

    We provide a commercial license for companies that are interested to have oauthd into their server keep close the modifications they have made on oauthd or the code they linked to oauthd.
    The prize of the commercial license depends on the support service required.
